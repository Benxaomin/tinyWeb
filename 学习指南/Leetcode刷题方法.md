# LeetCode 刷题方法

## 一、刷题策略（重思维，轻题量）—— 为什么这四步是最高效的？

**核心原理：这是基于"主动回忆"（Active Recall）和"间隔重复"（Spaced Repetition）的学习科学。**

### Step 1: 先自己思考15分钟（不看答案，手写思路）

**为什么要15分钟？**

- **5分钟**：读题+理解边界条件
- **10分钟**：进入"挣扎区"（Desirable Difficulty）。大脑在尝试建立问题模型，即使最终没解出来，这个"挣扎"过程也在构建神经连接
- **写什么？**：不写代码，写**思路框架**。比如"两数之和"：暴力法是O(n²)的双重循环，优化可能是哈希表O(n)。画出这个思维路径即可

**关键**：哪怕完全不会，也要写出"我觉得应该用XX方法，因为..."。这逼你暴露思维盲区。

### Step 2: 看题解，理解最优解（为什么这样写？时间/空间复杂度是多少？）

**不要直接看代码！先看"思路"部分。**

- **对比**：我的思路 vs 题解思路，差在哪一步？
  - 我没想到哈希表 → 说明我对"用空间换时间"的敏感度不够
  - 我想到哈希表但边界处理错了 → 说明我对边界条件的系统性思考不足
- **复杂度分析**：必须能脱口而出。这是面试必问，也是衡量你理解深度的标准。
  - 时间复杂度：O(n)还是O(n²)？为什么？
  - 空间复杂度：O(1)还是O(n)？能不能优化？

### Step 3: 自己独立写一遍（不看答案，完全自己敲）

**这是分水岭。90%的人失败在这里。**

- **操作**：关掉题解页面，打开IDE，像面试一样写。
- **卡住了怎么办？**：看一眼题解的关键一行（不要看全部），然后继续写。反复这个过程直到跑通。
- **为什么？**：看懂了≠会写了。编程是肌肉记忆+逻辑组织的结合，必须亲手敲才能内化。

### Step 4: 隔天复习（第二天开始新题前，先默写昨天的题）

**基于艾宾浩斯遗忘曲线。**

- **操作**：第二天打开LeetCode，不看昨天的代码，直接新建文件重写给定的输入。
- **标准**：如果能在5分钟内写完且通过，算掌握；如果卡壳超过10分钟，说明没真正理解，需要重新看题解。
- **效果**：这样刷30题，等于别人刷100题。因为每道题都经过了"学习→遗忘→提取"的完整循环。

## 二、如何结合现有学习（关键）—— 知识联结（Elaboration）策略

**核心原理：孤立的知识留存率<10%，与已有经验联结的知识留存率>90%。**

### 场景1：oslab进程调度 → 数据结构/算法

**oslab中你会遇到：**
- Linux 0.11的调度函数`schedule()`：遍历任务数组，找counter最大的就绪进程
- 任务切换时的现场保存/恢复

**对应LeetCode题目：**
- **621. 任务调度器**：理解"冷却时间"和"贪心"策略（为什么操作系统要有时间片？）
- **622. 设计循环队列**：进程就绪队列通常用循环队列实现（固定大小，避免内存分配开销）
- **295. 数据流的中位数**：使用**优先队列（堆）**实现。Linux CFS调度器用**红黑树**（平衡二叉搜索树）管理进程，原理类似：快速找到最小vruntime的进程

**结合点（明天就可以做）：**
当你看到`kernel/sched.c`中的`schedule()`函数时，不要只看注释，去LeetCode做一题"设计任务调度器"，你会突然理解：**原来调度就是解决"在给定约束下最大化吞吐量"的算法问题**。

### 场景2：tinyweb路由解析 → Trie树（前缀树）

**tinyweb中你会遇到：**
- URL路由匹配：`/api/users/:id` 如何匹配到对应的handler？
- 静态文件路径解析

**对应LeetCode题目：**
- **208. 实现 Trie (前缀树)**：路由匹配的底层数据结构
- **211. 添加与搜索单词 - 数据结构设计**：支持通配符`.`的匹配（类似路由中的`:id`参数）
- **677. 键值映射**：前缀统计（统计以某路径开头的请求数，用于限流）

**结合点（当你学到路由部分时）：**
Gin框架（Go的热门Web框架）的路由实现就是Trie树的变种。你先刷完Trie树的题目，再看tinyweb或Gin的源码，会看到：
- 每个节点存储一个路径段（如`api`、`users`）
- 叶子节点存储handler函数指针
- 匹配过程就是Trie树的遍历+回溯

**你会恍然大悟**：这不是什么"框架魔法"，就是数据结构的实际应用。

### 场景3：链表操作 → 内存管理思维

**tinyweb中你会遇到：**
- Session管理（链表存储活跃连接）
- 任务队列（待处理的请求）

**对应LeetCode题目：**
- **206. 反转链表**：理解指针操作的本质（`next`指针的重新指向）
- **141. 环形链表**：检测循环引用（内存泄漏排查的基础思维）
- **21. 合并两个有序链表**：多路归并的基础（排序算法中的merge步骤）

**结合点：**
当你用C写oslab或理解tinyweb的内存池时，链表操作的熟练度决定了你是否能写出**无内存泄漏**的代码。刷这些题不是浪费时间，是在训练**指针安全意识**。

## 三、具体执行建议（明天就能用）

**建立"双链笔记"系统：**

1. **在oslab/tinyweb学习中**：
   - 每当看到不理解的数据结构操作，记下：**"这里用了XX结构，我要去LeetCode验证"**
   - 例：看到`switch_to`宏涉及寄存器保存，记下："这类似栈的操作，我要刷一道'用栈实现队列'"

2. **在LeetCode刷题时**：
   - 每道题的笔记加一栏：**"实际应用场景"**
   - 例：做完Trie树，写："可用于路由匹配、自动补全、IP路由表"

3. **每周复盘**：
   - 回顾：我这周学的oslab代码，用到了哪些我刷过的算法题的数据结构？
   - 例：这周学了进程调度，发现用了优先队列 → 复习LeetCode上的堆相关题目

**这样，你的知识网络会越来越密，面试时提到一个点，你能连到三个面。**
